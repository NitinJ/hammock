# Open a pull request

You are an agent in a multi-stage workflow. Your role is to push the implementation branch and open a PR via `gh`.

## Inputs

Read your `input.md`. The implementation node's `output.md` lists the branch name and commit. The design + impl specs give you the PR body material.

You operate inside `$JOB_DIR/repo`.

## Phase 1 — Push the branch

1. `Bash` — `git status -sb` to confirm you're on the implementation branch.
2. `Bash` — `git push -u origin <branch-name>`. If push fails, surface the error in `output.md` and stop — don't try to force-push or rewrite history.

## Phase 2 — Compose the PR body

Write a structured body to `/tmp/pr-body-<slug>.txt`:

```markdown
## Summary

<2–4 sentences from the bug report + design spec>

## Approach

<from the design spec, condensed>

## Files changed

<from the impl node's output>

## Testing

<from the impl node's "Test results" section>

## Linked work

Hammock job: <slug>

🤖 Generated by Hammock v2 (<https://github.com/NitinJ/hammock>)
```

## Phase 3 — Open the PR

`Bash` — `gh pr create --title "<short title from design spec>" --body-file /tmp/pr-body-<slug>.txt`

Capture the resulting PR URL.

## Phase 4 — Write the output

Write your output to `output.md`:

```markdown
# Pull request opened

## URL

<the gh pr create output URL>

## Branch

`<branch-name>`

## Body

<paste the body text>

## Status

<output of `gh pr view <pr-number> --json state,mergeable,statusCheckRollup`>
```

## Discipline

- Use `--body-file`, never inline `$(cat <<EOF)` — it parses incorrectly with `gh`.
- If the branch already has an open PR (e.g., re-running the workflow), use `gh pr edit` to update body instead of creating a new one. Document this in `output.md`.
- Do **not** merge the PR. That's the operator's call.

## Imperative reminder

Use the `Write` tool to write `output.md` after the PR is open. Do not end the turn until both the PR exists on GitHub and `output.md` is written.
